## 一、数据来源与使用约定

### 1.1 数据来源

本课程统一使用 **AirROI 提供的 Airbnb 完整数据**，数据入口为：

> https://www.airroi.com/data-portal/

请同学们注意：

- 需遵守 AirROI 网站与数据使用条款，只在本课程教学与学习范围内使用；  
- 请勿将原始数据集公开上传到公开仓库或互联网上；  
- 本模板仓库中不包含真实数据文件，只在 `data/README.md` 中说明数据内容与路径。

### 1.2 本地数据组织方式与城市/时间选择

建议在你们自己的小组仓库中，将真实数据（或其抽样/预处理结果）放在 `data/` 目录下，例如：

```text
data/
  README.md          # 说明本组所用数据：来源、城市/时间范围、文件说明
  listings.csv       # 房源静态与历史表现
  calendar.csv       # 日历价格与可订情况（如需要）
  reviews.csv        # 评论数据（如需要）
```

**城市与时间范围约束：**

- 城市必须从 **Tokyo、Seoul、London、Paris、Rio de Janeiro** 中选择一个或多个；  
- 时间范围（例如近 12 个月、近 24 个月等）由各小组自行决定，不做统一限制；  
- 各小组需在 `data/README.md` 中清楚写明：  
  - 所选城市列表；  
  - 所选时间范围（如具体年份/月度、TTM/L90 等窗口）；  
  - 重要筛选条件（如房源类型、最少评价数等）。

> 提示：建议通过 `.gitignore` 忽略大体量原始数据文件，仅在仓库中保留 `data/README.md` 及必要的预处理脚本。

## 二、Airbnb / AirROI 课题总览

所有小组均使用 **同一份 AirROI 数据集**，但可从下表中 **选择不同课题** 完成。  
每个小组只需要完成一个课题（也可以在老师同意的前提下做适度扩展）。

### 2.1 通用要求

无论选择哪个课题，项目报告中都必须体现以下核心内容：

1. **完整的机器学习项目流程**
   - 数据理解与 EDA：分布、缺失值、异常值、相关性等；
   - 数据清洗与特征工程（含合理的特征选择或构造）；
   - 合理的 Train / Validation / Test 划分；
   - 至少 **2 个模型或方法的对比**（含 baseline）；
   - 误差分析与关键结果可视化；
   - 面向业务/运营的结论与建议，而不仅是指标罗列。

2. **时间与信息泄露控制**
   - 做“预测未来”时，不得使用包含未来信息的特征或聚合统计；
   - 预测某个指标（如 `ttm_revpar`）时，不得把该指标或其变形再作为输入特征；
   - 时间范围的具体选择（如近一年、近两年）不限制，但需在报告中说明。

3. **Git 协作与可重复性**
   - 每位成员需要有真实、持续的 commit 记录；
   - 仓库中应包含一份 **清晰的运行说明**（即本 README + 你们自己的补充说明）；
   - 报告中需附上简要的 `git log` 截图和成员分工说明。

### 2.2 课题列表与建议难度

> 难度仅供选题时参考，不影响成绩上限。

## 三、项目目录结构（模板）

本仓库提供了一个推荐的项目结构模板。建议你们在各自小组仓库中 **整体拷贝** 或据此搭建结构：

```text
data/                  # 本组使用的数据集（不上传原始大文件到公开仓库）
  README.md            # 描述数据来源、城市/时间范围与字段说明
notebooks/
  README.md            # Notebook 使用说明
  ...                  # 你们的 *.ipynb 文件（EDA、特征工程、模型原型等）
projects/
  project1_*.md        # 课题说明文档（只读）
  ...
reports/
  report_template.md   # 报告结构模板（复制一份作为你们最终报告草稿）
src/
  main.py              # 主程序入口：训练与评估（需要根据项目改写）
  utils.py             # 工具函数（数据读取、划分、评估等）
.gitignore
README.md              # 本说明文件，可以按本组实际情况适当补充
requirements.txt       # Python 依赖列表
```

> 提示：`projects/` 和 `report_template.md` 是 **课程统一提供** 的说明性文件，  
> 各小组在自己的仓库中 **不需要修改其内容**，只需要基于这些说明开展工作即可。

## 四、小组角色与内容分工建议（固定 4 人）

每个小组固定 4 名同学，建议采用以下角色划分（可部分重叠，但每位同学至少负责 1–2 个核心模块）：

1. **数据 & EDA 负责人（Data & EDA Lead）**
   - 负责原始数据下载、清洗与合并（listings / calendar / reviews 等）；  
   - 完成主要的 EDA：分布、缺失值、异常值、城市与时间选择说明；  
   - 与建模负责人沟通特征工程思路。

2. **特征工程 & 建模负责人（Modeling Lead）**
   - 负责设计 baseline 模型与改进模型（含超参数选择）；  
   - 主导特征选择/构造、模型训练与结果分析；  
   - 输出模型性能对比表格与关键可视化图。

3. **工程实现 & 代码结构负责人（MLOps / Engineering Lead）**
   - 负责整理 `src/` 目录结构与运行脚本，使项目可一键运行；  
   - 设计命令行参数、日志记录与结果保存方式；  
   - 维护 `requirements.txt` 与补充本 README 中“如何运行本项目”的具体说明。

4. **报告撰写 & 展示负责人（Report & Presentation Lead）**
   - 负责撰写项目报告初稿并协调其他同学补充内容；  
   - 负责图表美化、结构调整与最终版整理；  
   - 负责期末展示 PPT / 演示脚本准备。

> 建议：在上面的成员信息表中，给每位同学写明主要角色（例如“Data & EDA”“Modeling”“Engineering”“Report”），  
> 这样既有利于任务分工，也方便老师根据仓库记录与报告进行个人评分。

## 五、推荐开发流程（按周参考）

可参考如下时间线推进项目（实际安排以课堂说明为准）：

1. **第 1–2 周：开题与数据确认**
   - 小组成员确定、创建私有 GitHub 仓库；
   - 从本模板仓库复制目录结构与 `projects/` 课题说明；
   - 老师指定统一的 AirROI 数据集后，在 `data/README.md` 中记录：  
     - 数据来源链接、下载方式；  
     - 所选城市（Tokyo / Seoul / London / Paris / Rio de Janeiro 中的哪个/哪些）；  
     - 所选时间范围与其他筛选条件。

2. **第 2–4 周：EDA 与 baseline 模型**
   - 在 `notebooks/` 中完成数据探索与可视化；
   - 在 `src/` 中实现一个能跑通的 baseline（如简单回归/分类模型）；
   - 通过命令行运行 `main.py`，验证环境与数据流程是否正确。

3. **第 4–7 周：模型改进与实验设计**
   - 设计两种及以上模型或改进方案（含 baseline）；
   - 完成必要的对比或消融实验，记录超参数与结果；
   - 形成中期汇报用的图表与初步结论。

4. **第 7–9 周：结果分析与报告撰写**
   - 梳理实验结果，结合业务背景进行解释；
   - 按 `reports/report_template.md` 的结构撰写最终项目报告；
   - 自查是否满足“通用要求”和“Git 协作要求”。

## 六、运行示例（环境自测）

> 以下示例仅用于验证 Python 环境和代码结构是否能正常运行，  
> 实际项目中你们需要根据选题与数据替换/扩展相应代码。

1. 安装依赖（建议使用虚拟环境或 Conda）：

   ```bash
   pip install -r requirements.txt
   ```

2. 运行示例程序（例如 `src/main.py` 中的占位示例）：

   ```bash
   python src/main.py --model logistic --test-size 0.2
   ```

完成自己的项目后，建议你们的代码至少支持：

- 通过命令行指定数据路径、模型类型、关键超参数等；
- 在终端输出主要评估指标（如 Accuracy、F1、RMSE、MAE 等）；
- 在 `outputs/` 或 `reports/` 等目录下生成图表或结果文件（如有需要，可自建目录）。

## 七、GitHub 使用与协作要求

1. **小组仓库**
   - 每个小组一个 **私有仓库**，所有开发工作必须在该仓库中完成；
   - 老师会综合参考代码仓库记录与项目报告进行评分。

2. **Commit 规范**
   - 建议每位成员至少有 **10 个以上有效 commit**；
   - 提交信息简洁明了，例如：  
     - `feat: add xgboost model`  
     - `fix: handle missing values in calendar`  
     - `refactor: clean feature engineering pipeline`

3. **分支与合并（推荐）**
   - 为新功能创建分支，如 `feature/eda`, `feature/model-xgboost`；
   - 通过 Pull Request 合并到主分支，并在 PR 中简要说明变更内容；
   - 合并前由组内其他成员进行简单代码 Review。

4. **Issue / Projects（可选加分项）**
   - 使用 Issues 记录 Bug、改进建议和待办任务；
   - 使用 Projects 看板管理进度（如：To Do / Doing / Done）。

5. **报告中的协作说明**
   - 报告中需附一张 `git log`（或 GitHub Insights / Contributors）的截图；
   - 在报告末尾列出每位成员的主要贡献（代码、实验、报告撰写等）；  
   - 如发现贡献严重失衡，老师会在个人成绩中做相应调整。

## 八、评分参考（满分 100）

> 最终评分以课堂说明与课程大纲为准，以下为参考划分（与所选城市/时间窗口无关）：
