---

## 一、课程项目目标

通过一个完整的小型机器学习项目，让同学们：

1. 体验从 **选题 → 数据获取 → 建模 → 评估 → 报告** 的完整流程；  
2. 能够在 GitHub 上进行 **协同开发**，使用版本控制管理代码；  
3. 学会阅读与复用开源代码，能够在 baseline 上进行改进；  
4. 能够用图表和文字清晰表达自己的模型设计与实验结论。

---

## 二、项目选题范围（示例，可自拟）

每组需选择 **一个核心任务**，数据可以来自公开数据集或自建数据集。以下为建议方向：

1. **结构化表格数据建模**
   - 场景：信用评分、酒店评价打分预测、旅客行为分析等
   - 模型：Logistic Regression、SVM、Tree / RF / XGBoost 等

2. **图像分类或检索**
   - 数据示例：MNIST、CIFAR-10、公开商品/菜品/酒店图片等
   - 模型：CNN、ResNet 等（可使用 torchvision 等预训练模型）

3. **文本情感分析与文本分类**
   - 场景：点评情感极性、邮件垃圾识别、评论主题分类等
   - 模型：TF‑IDF + 传统 ML、RNN/CNN、预训练语言模型等

4. **推荐系统**
   - 场景：电影 / 商品 / 酒店 / 旅游线路推荐
   - 模型：协同过滤、矩阵分解、浅层神经网络等

5. **时间序列预测**
   - 场景：客流量、销量、流量、价格等随时间变化的预测
   - 模型：ARIMA、LSTM/GRU、基于树的序列建模等

6. **图数据与图神经网络（进阶，可选）**
   - 场景：社交网络、知识图谱的节点分类或链路预测
   - 模型：GCN、GraphSAGE 等

7. **简单强化学习任务（进阶，可选）**
   - 场景：迷宫导航、简单博弈、资源分配
   - 模型：Q‑learning、DQN 等

8. **大模型辅助的机器学习工作流（进阶，可选）**
   - 场景：使用大模型进行数据清洗、特征构造、数据增强，再训练传统模型
   - 要求：说明大模型介入前后效果变化与代价

> 自拟题目需提前与老师确认，确保数据规模、难度与课程要求匹配。

## 三、目录结构与开发流程

本模板推荐的目录结构：

```text
data/                  # 放数据集（或在 README 说明数据下载方式）
src/
  main.py              # 主程序入口：训练与评估（需要根据项目改写）
  utils.py             # 工具函数（如读数据、划分数据集等）
notebooks/
  README.md            # Notebook 使用说明
  ...                  # 你们的 *.ipynb 文件
reports/
  report_template.md   # 报告结构模板（建议复制后填写）
requirements.txt       # Python 依赖列表
```

建议开发流程：

1. **开题与数据确认（第 1–2 周）**
   - 选定课题与数据源，在本 README 顶部补全项目信息；
   - 在 `data/README.md` 中写明数据集来源与基本情况。

2. **数据探索与 baseline（第 2–4 周）**
   - 使用 `notebooks/` 目录下的 Notebook 做数据探索（EDA）；
   - 在 `src/` 中实现一个可运行的 baseline，并能通过命令行运行。

3. **模型改进与实验设计（第 4–7 周）**
   - 设计两种及以上模型或改进方案；
   - 完成 ablation 或对比实验，记录结果。

4. **结果分析与报告撰写（第 7–9 周）**
   - 整理实验结果，完成图表与文字分析；
   - 依据 `reports/report_template.md` 撰写项目报告。

## 四、运行方式（示例）

> **注意：当前 `src/main.py` 和 `src/utils.py` 仅给出 iris 数据集的示例代码，用于验证环境是否配置成功。完成自己的项目时，必须根据实际数据与任务修改 / 替换这些代码。**

1. 安装依赖（建议使用虚拟环境或 conda）：

   ```bash
   pip install -r requirements.txt
   ```

2. 运行示例程序（iris baseline）：

   ```bash
   python src/main.py --model logistic --test-size 0.2
   ```

3. 完成自己的项目后，应该能够：

   - 指定数据路径、模型类型、超参数等命令行参数；
   - 在终端输出主要指标（如 Accuracy、F1、RMSE 等）；
   - 在 `outputs/` 或 `reports/` 下生成关键图表或结果文件（如有需要，可自建目录）。

## 五、GitHub 使用与协作要求

1. **小组仓库**：每个小组对应一个私有仓库，所有开发工作必须在该仓库中完成。  
2. **Commit 规范**：
   - 建议每位成员至少有 **10 个以上有效 commit**；
   - 提交信息使用简短英文或中英文，例如：`feat: add xgboost model`、`fix: data split bug`。

3. **分支与合并（建议）**：
   - 可以为新功能创建分支，如 `feature/model-xgboost`；
   - 通过 Pull Request 合并到主分支，并在 PR 中简要说明变更内容。

4. **Issue / Projects（可选加分）**：
   - 使用 Issues 记录待办任务与 bug；
   - 使用 Projects 看板管理进度。

老师在评分时会综合考虑 **代码质量、目录结构、commit 记录和协作方式**。

## 六、评分细则示例（满分 100）

> 最终评分以老师课堂说明为准，以下为参考划分：

1. **项目设计与数据选择（15 分）**
   - 问题定义清晰，数据来源合理、可靠；
   - 数据预处理规范，有基本统计分析。

2. **实现与代码质量（30 分）**
   - 代码能顺利运行，无明显逻辑错误；
   - 结构清晰，函数划分合理，有必要的注释与 README；
   - 合理使用 Git 进行版本控制。

3. **模型与实验设计（25 分）**
   - 至少两种模型或改进方案；
   - 实验设置合理，有对比与消融；
   - 指标选择恰当。

4. **结果分析与报告（15 分）**
   - 报告结构清晰，图表规范；
   - 能够结合结果进行有意义的分析，而不仅是罗列数字。

5. **小组协作与个人贡献（15 分）**
   - 每位成员在代码、报告或实验中有实际贡献；
   - 通过 commit 记录、`reports` 中的分工说明以及组内互评进行综合评定。
